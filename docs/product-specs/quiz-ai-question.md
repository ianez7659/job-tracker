# Daily Interview Quiz — AI Card-Based Question (Step 3)

## Purpose

One of the 5 daily quiz questions is personalized based on the user's active job card.
This question is generated by OpenAI using the job card context (company, role, stage, JD).
It is harder and more specific than the pool-based questions.

If generation fails for any reason, the session falls back to an extra exact-category question automatically. Users never see an error.

---

## OpenAI Integration

- **Model**: `gpt-4o-mini`
- **Response format**: `response_format: { type: "json_schema", json_schema: { strict: true, schema: {...} } }`
- **Max tokens**: 800
- **Temperature**: 0.7
- **Timeout**: 20 seconds

### Why json_schema over json_object?

`json_schema` with `strict: true` enforces the exact output structure at the model level, reducing the chance of malformed responses compared to `json_object`. A separate server-side validation layer provides an additional safety net.

---

## Prompt Structure

### System prompt

The system prompt instructs the model to:
- Output valid JSON only (no markdown)
- Generate exactly one English multiple-choice question
- Use ids `a`, `b`, `c`, `d` for choices
- Include `correctChoiceId`, `correctExplanation`, `wrongExplanations` for all 3 incorrect choices
- Set difficulty to `"medium"` or `"hard"`
- Base the question on the provided job card context

### User prompt

Built from `AiQuestionSlotInput`:
- `jobTitle` — the role title
- `jobCompany` — company name
- `jobStage` — mapped to human-readable label (e.g. "interview1" → "first interview")
- `userCategory` — underscore format, displayed with spaces
- `jd` — first 2,000 chars of the job description if available

---

## Output JSON Schema

```json
{
  "question": "string",
  "choices": [
    { "id": "a" | "b" | "c" | "d", "text": "string" }
  ],
  "correctChoiceId": "a" | "b" | "c" | "d",
  "correctExplanation": "string",
  "wrongExplanations": [
    { "choiceId": "a" | "b" | "c" | "d", "explanation": "string" }
  ],
  "difficulty": "medium" | "hard",
  "tags": ["string"]
}
```

`minItems`/`maxItems` are not used in the schema (not supported in strict mode).
Count validation is enforced server-side.

---

## Server-Side Validation

Performed after OpenAI responds, independent of the structured output guarantee.

| Field | Rule |
|-------|------|
| `question` | Non-empty string |
| `choices` | Exactly 4 items; ids must be `a/b/c/d` with no duplicates; each `text` non-empty |
| `correctChoiceId` | One of `a/b/c/d`; must exist in choices |
| `correctExplanation` | Non-empty string |
| `wrongExplanations` | Exactly 3 items; each `choiceId` is one of `a/b/c/d` and NOT the correct choice; each `explanation` non-empty |
| `difficulty` | `"medium"` or `"hard"` only (never `"easy"`) |
| `tags` | Array of strings (may be empty) |

Validation returns all errors found. Any error → `null` returned → fallback activates.

---

## Fallback Behavior

If `generateAiQuizQuestion()` returns `null` (for any reason), the session engine automatically uses the without-card composition:

| Slot | Source | Count |
|------|--------|-------|
| Common | `category = "common"` | 2 |
| Group-common | e.g. `"tech-common"` | 1 |
| Exact category | e.g. `"web-development"` | **2** |

No user-visible error. The session is created with 5 questions regardless.

Reasons that trigger fallback:
- `OPENAI_API_KEY` not set
- OpenAI API error or network failure
- Response timeout (> 20 seconds)
- Empty or unparseable response
- Server-side validation failure

---

## Cost & Retry Prevention

- AI is called **once per session creation only**.
- The result (question snapshot or fallback) is committed to `DailyQuizSessionItem`.
- On page reload, the existing session is returned — AI is never re-called.
- If generation failed and fallback was used, the same day does NOT retry AI (the snapshot already exists with 5 fallback questions).
- A new calendar day creates a new session, which gets a fresh AI attempt.

---

## Snapshot Storage

When AI generation succeeds, the question is stored as a `DailyQuizSessionItem` with:

| Field | Value |
|-------|-------|
| `questionId` | `null` (not in QuizQuestion bank) |
| `sourceType` | `"ai_card_based"` |
| `renderedQuestionSnapshot` | AI-generated question text |
| `choicesSnapshot` | Shuffled choices (Fisher-Yates) |
| `correctChoiceIdSnapshot` | From AI response |
| `correctIndexSnapshot` | Computed after shuffle |
| `correctExplanationSnapshot` | From AI response |
| `wrongExplanationsSnapshot` | From AI response |
| `metadata` | `{ source, jobCardId, sourceId, generationStatus: "success" }` |

---

## Monitoring

AI question quality should be reviewed periodically in production:
- Query `DailyQuizSessionItem` where `sourceType = "ai_card_based"` for generated questions
- Check `isCorrect` distribution — if users always get AI questions wrong, the difficulty may be too high
- Review `metadata.generationStatus` for failure rates
- Consider logging detailed errors server-side for debugging

---

## Environment Variables

| Variable | Required | Purpose |
|----------|----------|---------|
| `OPENAI_API_KEY` | Yes | OpenAI API authentication |

If `OPENAI_API_KEY` is not set, AI question generation is silently skipped and fallback activates. No crash, no user-visible error.
